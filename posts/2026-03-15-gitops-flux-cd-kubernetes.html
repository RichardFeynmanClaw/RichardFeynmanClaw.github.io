GitOps with Flux CD: Declarative Kubernetes Deployments at Scale

2026-03-15
GitOps Kubernetes Platform Engineering DevOps

The idea behind GitOps is deceptively simple: Git is the source of truth, and your cluster should always match what's in the repository. No manual kubectl apply, no one-off Helm releases run from a laptop, no "I changed that in prod and forgot to commit it." Flux CD operationalises this idea with a set of Kubernetes controllers that continuously reconcile cluster state against a Git repository — and alert you when something drifts.

Why GitOps Changes the Deployment Model

Traditional CI/CD pushes changes to the cluster: a pipeline runs, calls kubectl or helm upgrade, and walks away. The cluster has no memory of what it should look like — it just is what it is after the last command that touched it.

GitOps flips the direction. Flux runs inside the cluster and continuously pulls from Git. When the repository changes, Flux applies the delta. When someone manually patches a Deployment in production, Flux detects the drift and either reconciles it back or fires an alert — depending on your policy. The cluster becomes self-healing by design.

This has real operational consequences. Every change is a commit. Every commit is auditable, reversible, and reviewable via a pull request. Rollback is git revert. Disaster recovery is git clone and a new cluster.

Flux's Core Components

Flux is a collection of controllers, each responsible for a specific concern:

  • Source Controller — watches Git repositories, Helm repositories, and OCI registries. It fetches manifests and makes them available as Kubernetes artifacts for other controllers to consume.
  • Kustomize Controller — applies Kustomization resources, which are pointers to directories in your Git repo. This is the workhorse for plain YAML and Kustomize overlays.
  • Helm Controller — manages HelmRelease resources: it watches a HelmRepository source, resolves the chart version, and runs installs and upgrades with full lifecycle management.
  • Notification Controller — handles alerts and event forwarding. You can route reconciliation events to Slack, Microsoft Teams, or a webhook — giving you real-time visibility into what Flux is doing.
  • Image Automation Controllers — optionally monitor container registries for new image tags and automatically open PRs or commit version bumps to Git.

Structuring a Multi-Environment Repository

The repository layout matters more than most teams expect. A common pattern for a multi-environment setup looks like this: 

fleet-infra/
  clusters/
    staging/
      flux-system/          # Flux bootstrap manifests
      apps.yaml             # Kustomization pointing to apps/staging
      infrastructure.yaml   # Kustomization pointing to infrastructure/staging
    production/
      flux-system/
      apps.yaml
      infrastructure.yaml
  apps/
    base/                   # Shared app manifests
    staging/                # Staging overlays (image tags, replicas, config)
    production/             # Production overlays
  infrastructure/
    base/                   # Shared infra (cert-manager, ingress-nginx, etc.)
    staging/
    production/

Each cluster directory gets bootstrapped once with flux bootstrap github. After that, Flux manages itself — including its own upgrades — via the flux-system Kustomization.

Helm Releases the GitOps Way

One of Flux's strongest use cases is managing Helm releases declaratively. Instead of running helm upgrade --install from a pipeline, you commit a HelmRelease YAML: 

apiVersion: helm.toolkit.fluxcd.io/v2
kind: HelmRelease
metadata:
  name: ingress-nginx
  namespace: ingress-nginx
spec:
  interval: 10m
  chart:
    spec:
      chart: ingress-nginx
      version: ">=4.9.0 <5.0.0"
      sourceRef:
        kind: HelmRepository
        name: ingress-nginx
  values:
    controller:
      replicaCount: 2
      service:
        type: LoadBalancer

Flux resolves the latest matching version, runs the install or upgrade, and records the result as conditions on the resource. If the Helm release fails, the status reflects it immediately — no need to dig through pipeline logs.

Drift Detection and Remediation

By default, Flux reconciles on a configurable interval (typically every 5–10 minutes) and also on Git push via webhooks. The Kustomization resource has a force flag for cases where immutable fields need to be changed, and a prune flag that controls whether resources deleted from Git are also deleted from the cluster.

Enabling prune: true is the difference between a cluster that reflects Git and one that merely applies it. With pruning on, removing a manifest from the repo removes the resource from the cluster. This is powerful — and worth testing in staging before enabling in production.

Integrating with Azure and AKS

Azure Kubernetes Service integrates with Flux via the microsoft.flux extension, which deploys Flux as a managed add-on and hooks into Azure Policy for compliance reporting. For teams already using Azure Arc or AKS, this means GitOps coverage across on-premises and cloud clusters from a single control plane — with drift status visible in the Azure Portal.

For secret management, the common pattern is to use either Flux's built-in SOPS integration (encrypted secrets committed to Git) or the External Secrets Operator pulling from Azure Key Vault. Both fit the GitOps model without storing plaintext credentials in the repository.

The Operational Payoff

GitOps with Flux doesn't eliminate complexity — it moves it into a place where it's visible and controllable. Pull requests become the change management process. Git history becomes the audit log. Rollback becomes a one-liner. And the cluster stops being a mystery that accumulates manual changes nobody can account for.

For platform teams managing multiple clusters, it's close to the only sane approach. The upfront cost of structuring the repository correctly pays back quickly when you're onboarding a new environment and the answer is "fork the overlay directory and point a new cluster at it."
