OpenTelemetry in Practice: Unified Observability for Cloud-Native Stacks

2026-03-12
OpenTelemetry Observability Kubernetes DevOps

For years, observability in Kubernetes meant duct-taping together three separate ecosystems: Prometheus for metrics, something like Jaeger or Zipkin for traces, and Fluentd or Loki for logs. Each had its own agents, its own config, its own query language. OpenTelemetry (OTel) didn't solve everything overnight, but it did something important: it standardised the instrumentation layer so you only have to solve that problem once.

What OpenTelemetry Actually Is

OpenTelemetry is a CNCF graduated project that defines a vendor-neutral API, SDK, and wire protocol (OTLP) for telemetry data — traces, metrics, and logs. It's not a backend. It doesn't store data or render dashboards. What it does is give you a single, consistent way to emit telemetry from your applications and infrastructure, then route it to whatever backend you choose: Grafana, Datadog, Honeycomb, Jaeger, Azure Monitor, or anything else that speaks OTLP.

The architecture has three main pieces: the SDK (embedded in your application code), the Collector (a sidecar or standalone daemon that receives, processes, and exports telemetry), and the backends that consume it. The Collector is where most of the operational leverage sits — it can filter, sample, transform, and fan out data to multiple destinations without touching application code.

Instrumenting Kubernetes Workloads

Auto-instrumentation is the fastest path to value. The OpenTelemetry Operator for Kubernetes lets you annotate a namespace or pod, and it injects the OTel SDK automatically — no code changes required. For Java, Python, Node.js, and .NET, this gives you traces and basic metrics for free at the HTTP and database layer. 

apiVersion: opentelemetry.io/v1alpha1
kind: Instrumentation
metadata:
  name: my-instrumentation
spec:
  exporter:
    endpoint: http://otel-collector:4317
  propagators:
    - tracecontext
    - baggage
  sampler:
    type: parentbased_traceidratio
    argument: "0.25"
  python:
    env:
      - name: OTEL_LOG_LEVEL
        value: warn

Apply this to a namespace and annotate your deployment with instrumentation.opentelemetry.io/inject-python: "true". The Operator handles the rest. For more control — custom span names, business-level attributes, manual trace context propagation — you reach into the SDK directly, but the auto-instrumentation baseline is surprisingly complete.

The Collector: More Than a Proxy

Running the OTel Collector as a DaemonSet means every node has a local receiver. Applications send telemetry to localhost, which minimises latency and decouples your code from backend topology. The Collector then handles batching, retry logic, and routing.

The processor pipeline is where things get interesting. A few patterns worth knowing:

  • Tail-based sampling: Buffer traces until they complete, then sample based on outcome (e.g., keep all errors, sample 5% of successful requests). This is fundamentally better than head-based sampling for finding problems.
  • Attribute filtering: Strip PII or high-cardinality labels before they hit your backend. Cheaper storage, cleaner data.
  • Span metrics connector: Generate RED metrics (Rate, Errors, Duration) directly from trace data. You get a consistent metrics layer without separate instrumentation.
  • Multi-backend fan-out: Send traces to Jaeger for local debugging and Datadog for production alerting simultaneously. Change backends without redeploying applications.

Correlating Signals

The real payoff from OTel comes when traces, metrics, and logs share the same context. A trace ID injected into log output means you can pivot from a slow Prometheus alert to the exact request trace that explains why — without guessing which pod or which timestamp to look at.

In practice, this means configuring your logging library to read the current OTel span context and attach trace_id and span_id as structured log fields. Grafana can then link from a Loki log entry directly to the corresponding Tempo trace. One click, full context.

Where the Complexity Hides

OTel's abstraction is clean, but the operational surface area is real. The Collector configuration grows fast — multiple receivers, processors, exporters, and pipelines can combine in non-obvious ways. Cardinality in metrics is still your problem: OTel doesn't prevent you from creating ten million unique label combinations, it just makes it easier to do at scale.

Schema stability matters too. Semantic conventions (the standard attribute names for HTTP status codes, database systems, cloud resources) changed significantly between OTel spec versions. If you're upgrading an existing deployment, check for breaking changes in attribute names before you assume your dashboards still work.

The Bottom Line

OpenTelemetry is now the right default for instrumentation in cloud-native stacks. The vendor lock-in argument alone justifies the adoption — you can swap backends as requirements change without touching application code. Add the Operator for auto-instrumentation, deploy the Collector as a DaemonSet, and you have a solid observability foundation that scales with your platform. Fine-tuning sampling rates and building the correlation layer between signals takes time, but the baseline value arrives quickly.
