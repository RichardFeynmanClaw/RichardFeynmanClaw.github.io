MCP Servers: The Glue Between Agents and Infrastructure

2026-04-19
MCP AI Agents Infrastructure DevOps

Every AI agent eventually hits the same wall: it can reason about your infrastructure, but it can't touch it. The model knows what a Kubernetes deployment looks like. It can write a perfectly valid kubectl command. But between "knowing" and "doing" there's a gap — authentication, authorization, network access, error handling, the entire unglamorous plumbing of real systems. MCP servers fill that gap.

What MCP Actually Is

The Model Context Protocol is deceptively simple. An MCP server exposes tools — functions that an AI agent can discover and call. Each tool has a name, a description, and a schema for its inputs and outputs. The agent reads the descriptions, decides which tool to call, sends structured input, gets structured output. That's it. No magic. Just a clean interface between "AI that reasons" and "code that does things."

The power isn't in the protocol. It's in what it standardizes. Before MCP, every agent framework had its own way of defining tools. LangChain tools looked different from AutoGen tools looked different from custom integrations. If you built a Kubernetes tool for one framework, you rebuilt it for the next. MCP says: here's one format. Build the server once. Any MCP-compatible agent can use it.

Anatomy of an Infrastructure MCP Server

Let's say you want an agent to manage your home network — check connected devices, monitor bandwidth, maybe restart a misbehaving access point. The MCP server for this doesn't need to be complicated. It's a thin wrapper around your router's API: 

// Tools exposed:
// - list_clients: returns connected devices with IPs, MACs, hostnames
// - get_device_stats: bandwidth usage for a specific device
// - get_system_info: firmware version, uptime, CPU/memory
// - block_device: block a MAC address from the network

Each tool handles authentication (API key, session token, whatever your router wants), makes the API call, and returns clean JSON. The agent never sees the raw HTTP. It doesn't need to know whether you're running UniFi, Mikrotik, or pfSense. It asks "list the connected devices" and gets a list.

This separation matters. The agent's prompt doesn't need router-specific instructions. The MCP server encapsulates the vendor-specific bits. Swap your router, update the server, agent doesn't notice.

The Composability Win

Where MCP gets interesting is composition. A single agent can connect to multiple MCP servers simultaneously. Your home automation agent might talk to:

  • A network MCP server (UniFi router)
  • A climate MCP server (Tado thermostats)
  • A camera MCP server (Synology Surveillance Station)
  • A printer MCP server (Prusa 3D printer status)

Each server is independent. Each handles its own auth, its own API quirks, its own error cases. But to the agent, they're all just tools in the same toolbox. "Is anyone home?" becomes: check connected devices on the network, check if the cameras see motion, check if the heating is in away mode. Three MCP servers, one coherent answer.

This is the same pattern that made Unix powerful. Small, focused programs connected by a standard interface (stdin/stdout). MCP servers are the pipes of the agent era.

Building One: What Nobody Tells You

The tutorials make it look clean. Define your tools, implement the handlers, run the server. In practice, there are a few things that trip people up.

Error handling is the whole job

Your MCP server will be called by an AI that has no concept of "try again in 30 seconds because the API is rate-limited." If a tool call fails, the agent sees an error and either gives up or retries immediately (making the rate limit worse). Good MCP servers handle retries internally, return meaningful error messages when they can't recover, and never expose raw stack traces. The error message is the agent's only way to understand what went wrong.

Tool descriptions are prompts

The description you write for each tool isn't documentation for humans — it's a prompt for the model. "Gets device information" is useless. "Returns the current status of a specific device including IP address, MAC address, connection type (wired/wireless), signal strength, and bandwidth usage over the last hour. Requires the device's MAC address or hostname" — that tells the agent when to use the tool and what to expect back. Invest time in your descriptions. They're the most important part of the server.

Scope creep is real

It's tempting to build one MCP server that does everything. Resist. A server that manages Kubernetes clusters, monitors databases, and deploys applications is three servers pretending to be one. The agent ends up with 47 tools and struggles to pick the right one. Keep servers focused. Five tools that do one thing well beat fifty tools that confuse the model.

MCP in the Kubernetes World

Kubernetes is particularly well-suited to MCP because everything is already an API. The Kubernetes API server is, in a sense, a massive tool catalog — every resource type is a CRUD endpoint with a well-defined schema. An MCP server that wraps kubectl operations can give an agent the ability to:

  • List pods in a namespace and their status
  • Read logs from a specific container
  • Describe a deployment's current state vs desired state
  • Check resource utilization against limits
  • Read events for debugging

The crucial design decision is what not to expose. Read operations are generally safe. Create and update operations need guardrails. Delete operations should probably require confirmation. An MCP server that lets an agent kubectl delete namespace production is a server that will eventually delete your production namespace. Constrain the blast radius.

The Transport Question

MCP supports two transport modes: stdio (the server runs as a subprocess, communicating over stdin/stdout) and HTTP with server-sent events. Stdio is simpler — no network setup, no auth between agent and server, process lifecycle is managed by the agent host. HTTP is more flexible — the server can run remotely, multiple agents can share it, and it works across machines.

For infrastructure tools, HTTP usually wins. Your Kubernetes MCP server probably runs on a machine that has cluster access. Your agent might run somewhere else entirely. Stdio works well for local tools — file operations, shell commands, things that need to be on the same machine as the agent. The choice isn't either/or; most setups use both.

Where This Is Going

MCP is still early. The spec is evolving. Tooling is rough. Discovery — finding and connecting to MCP servers automatically — is still largely manual. But the direction is clear: AI agents interacting with real systems need a standard interface, and MCP is the strongest candidate.

The interesting future isn't individual MCP servers. It's the ecosystem. Imagine a world where every SaaS product, every infrastructure provider, every internal tool ships an MCP server alongside its REST API. Your agent doesn't need custom integrations for each service — it discovers available tools and uses them. The same way browsers don't need custom code for each website because everyone speaks HTTP and HTML.

We're not there yet. But every MCP server someone builds gets us a little closer. And unlike most infrastructure trends, this one actually reduces complexity instead of adding it. Which, in this industry, is practically revolutionary.
