MCP Servers: How AI Agents Grow Hands

2026-03-17
MCP AI Agents DevOps Automation Platform Engineering

A language model on its own is a read-only thing. It can reason about infrastructure, draft a Kubernetes manifest, explain a failing pod log — but it cannot do anything. The Model Context Protocol (MCP) changes that. It gives AI agents a structured way to call external tools, read live data, and take action — without the LLM provider needing to know anything about your specific stack.

What MCP Actually Is

MCP is an open protocol, originally published by Anthropic, that defines how an AI host (the thing running the model) communicates with an MCP server (the thing that knows how to talk to your infrastructure). The host sends JSON-RPC messages; the server responds with structured results. The model sees those results as context and decides what to do next.

The protocol defines three primitive types that servers can expose:

  • Tools — callable functions with typed input schemas. The model decides when to call a tool and with what arguments. Think kubectl_get_pods, run_terraform_plan, or query_prometheus.
  • Resources — structured data the model can read on demand: file contents, API responses, database records. Resources are passive; the model pulls them rather than the server pushing them.
  • Prompts — reusable, parameterised prompt templates that server authors can surface to the host. Useful for encoding domain-specific reasoning patterns.

Transport is deliberately simple. Servers run either over stdio (the host spawns the server as a child process) or over HTTP with Server-Sent Events. Both options work on a laptop, in a container, or across a network boundary.

Why This Matters for Platform Teams

Before MCP, integrating an AI assistant with internal tooling meant custom code for every integration, bespoke function-calling schemas per model provider, and no standard way to share or reuse tool definitions. Each team reimplemented the same patterns.

MCP standardises the interface. An MCP server for your internal Kubernetes management API can be used with any MCP-compatible host — today that means Claude, Copilot, Cursor, and a growing list of others. Write the server once; the tool is available everywhere the protocol is supported.

For platform teams, this unlocks a practical pattern: wrap your internal CLIs and APIs in MCP servers, deploy them alongside your AI gateway, and let operators use natural language to interact with systems they previously needed tribal knowledge to navigate. The model handles the translation; the MCP server enforces the actual API contract.

Building a Minimal MCP Server

A working MCP server for a simple Kubernetes tool can be written in under 100 lines of Python using the official SDK. The key pieces are tool registration and handler implementation: 

from mcp.server.fastmcp import FastMCP
import subprocess, json

mcp = FastMCP("k8s-tools")

@mcp.tool()
def get_pods(namespace: str = "default") -> str:
    """List pods in a Kubernetes namespace."""
    result = subprocess.run(
        ["kubectl", "get", "pods", "-n", namespace, "-o", "json"],
        capture_output=True, text=True, timeout=10
    )
    if result.returncode != 0:
        return f"Error: {result.stderr}"
    pods = json.loads(result.stdout)
    return "
".join(
        f"{p['metadata']['name']} — {p['status']['phase']}"
        for p in pods.get("items", [])
    )

if __name__ == "__main__":
    mcp.run(transport="stdio")

The FastMCP wrapper handles JSON-RPC serialisation, capability negotiation, and the transport layer. The tool function itself is just Python: call kubectl, parse the output, return a string. The model receives that string as a tool result and continues reasoning.

Deployment Patterns

For local development, stdio transport is the right choice — the host process spawns the server, communication is in-process, and there is nothing to configure network-wise. For production deployments where multiple agents or users share the same tools, HTTP transport with SSE is the better option: deploy the MCP server as a standard container, put it behind your existing auth layer, and point your AI gateway at it.

OpenClaw handles MCP server connections via its mcporter integration, which supports both transport modes and handles connection lifecycle, tool discovery, and result formatting automatically. Adding a new MCP server is a config entry; the tools it exposes appear in the agent's available tool set on the next connection.

Security Considerations

MCP servers execute real commands with real consequences. A tool that runs kubectl delete will delete things. This makes trust boundaries critical:

  • Scope tool permissions to the minimum required — a read-only kubeconfig for monitoring tools, a namespace-scoped service account for deployment tools.
  • Validate tool inputs before passing them to shell commands. The model will generally provide sensible arguments, but injection via crafted user prompts is a real attack surface.
  • Log tool invocations with their arguments. When an AI agent takes an action via MCP, that action should appear in your audit trail just like a human action would.
  • For destructive operations, consider a confirmation step in the tool implementation: return a dry-run summary first and require an explicit confirm=True argument to proceed.

The Bigger Picture

MCP is infrastructure-level plumbing for AI agents — not exciting in itself, but the thing that makes everything else possible. When an AI agent can reliably query your Prometheus instance, apply a Helm chart, or open a GitHub issue, the interaction model shifts from "AI that advises" to "AI that acts."

The protocol is still maturing, and the tooling ecosystem is moving fast. But the core design — a clean separation between the model and its tools, with a standard wire protocol between them — is solid. Platform teams that invest in building good MCP servers today are building infrastructure that will work with whatever AI host comes next.
