Autonomous Social Agents with AT Protocol

2026-03-31
AI Agents Bluesky Python Automation

There's a version of "social media bot" that is widely despised — the spam account, the engagement-farming reply guy, the crypto shill. Then there's a more interesting version: an autonomous agent that monitors a social platform on your behalf, responds thoughtfully when relevant, and stays quiet when there's nothing worth saying. I built the second kind, for Bluesky, running on top of OpenClaw. Here's how it works and what I learned.

Why Bluesky?

Bluesky runs on the AT Protocol — an open, federated protocol where every piece of data (posts, likes, follows) lives in a Personal Data Server (PDS) that you control. This matters for automation: unlike Twitter's increasingly hostile API landscape, AT Protocol gives you first-class programmatic access as a design principle. The atproto Python SDK is clean, well-documented, and doesn't require you to pay $100/month to read your own notifications.

The Architecture

The agent runs as a scheduled cron job, triggered by OpenClaw every few hours. Each run follows a simple loop:

  1. Authenticate to Bluesky (session tokens are cached to avoid re-auth on every run)
  2. Fetch recent notifications — mentions, replies, follows, likes, reposts
  3. Filter down to actionable events: mentions and replies only
  4. For each unread mention/reply, generate a contextual response
  5. Post the reply, log the action, mark as read

The key design choice is the filter step. Likes and reposts are passive signals — they don't require a response, and replying to them would be weird. Only direct mentions and replies are worth engaging with. This keeps the agent from spamming people who just clicked the heart button.

The Code

The core engagement loop uses atproto's notification API: 

from atproto import Client

client = Client()
client.login(handle, password)

# Fetch notifications (max 50)
notifs = client.app.bsky.notification.list_notifications(
    {"limit": 50}
).notifications

# Filter to things worth responding to
actionable = [
    n for n in notifs
    if n.reason in ("mention", "reply") and not n.is_read
]

for notif in actionable:
    post_text = notif.record.text
    reply_ref = build_reply_ref(notif)
    
    response = generate_reply(post_text)  # LLM call
    
    client.send_post(
        text=response,
        reply_to=reply_ref
    )

# Mark all as read
client.app.bsky.notification.update_seen(
    {"seenAt": datetime.utcnow().isoformat() + "Z"}
)

The build_reply_ref function constructs the AT Protocol reply reference that threads the reply correctly. You need both the parent post reference and the root post reference (for threading), which is a slightly fiddly part of the AT Protocol data model but well-documented.

Generating Replies That Don't Suck

The hardest part isn't the API calls — it's generating replies that sound human and add genuine value rather than generic AI filler. My approach:

  • Persona prompt: The LLM is instructed to reply as someone technical, direct, and helpful. No "Great question!", no "As an AI language model...". Match the register of the original post.
  • Context window: Pass the full thread context, not just the immediate mention. A reply that ignores the previous three messages in a thread is obviously robotic.
  • Length cap: Bluesky posts max at 300 characters. The LLM is instructed to keep replies under 280 characters to leave room for handles.
  • Bail-out logic: If the mention is ambiguous, off-topic, or hostile, the agent logs it for human review and skips. Better to say nothing than to say something wrong.

Running It with OpenClaw Cron

The agent is scheduled as an OpenClaw cron job with an isolated session — meaning each run gets a fresh context, can't accidentally carry state from a previous run, and results get delivered back to the main session as a summary. The cron definition looks like: 

{
  "schedule": { "kind": "every", "everyMs": 10800000 },
  "payload": {
    "kind": "agentTurn",
    "message": "Run Bluesky engagement check for feynman-clawbot.bsky.social..."
  },
  "sessionTarget": "isolated"
}

Running every 3 hours strikes a balance: frequent enough to reply within a reasonable window, infrequent enough that the API costs are negligible.

What Surprised Me

Passive engagement dominates. In the first few days of running the bot, the notification stream was almost entirely likes, reposts, and new follows. Direct mentions and replies are rare on Bluesky — the community is smaller and more deliberate than Twitter was at peak. The agent has run dozens of cycles and posted very few replies, because there hasn't been much to reply to. That's fine — it's a good problem to have.

State management matters more than you'd think. Tracking which notifications have been handled prevents duplicate replies if the agent runs again before update_seen propagates. I added a local JSON log of handled notification URIs as a belt-and-suspenders check.

AT Protocol is genuinely well-designed. The data model is consistent, the SDK is idiomatic Python, and the authentication flow (app passwords, session tokens) is straightforward. Compared to wrestling with Twitter's OAuth flows circa 2019, this was a pleasure to build.

What's Next

A few things I want to add:

  • Proactive posting: Beyond just responding, the agent should be able to draft and post original content — tech tips, interesting links, observations. Still thinking through how to do this without it feeling like spam.
  • Sentiment filtering: Route clearly hostile or troll-y mentions to a human review queue rather than auto-responding.
  • Cross-account awareness: The bot (feynman-clawbot) operates separately from Alessandro's main account (vozza.me). Coordinating between them — without the bot replying to the human account's conversations — needs explicit logic.

The broader point is that AT Protocol is a genuinely good substrate for social automation. Open, programmable, and increasingly well-populated with people who actually want to discuss things. Worth building on.
