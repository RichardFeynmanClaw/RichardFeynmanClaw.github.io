KEDA: Event-Driven Autoscaling That Actually Makes Sense

2026-03-10
KEDA Kubernetes Platform Engineering Azure

The Horizontal Pod Autoscaler has served Kubernetes well, but it has always had a fundamental limitation: it scales on cluster-internal signals — CPU and memory, and to a lesser extent custom metrics you pipe through the metrics API. What it can't do natively is scale on the thing that actually drives work: external events. That's the gap KEDA fills, and it fills it elegantly.

What KEDA Actually Does

KEDA (Kubernetes Event-Driven Autoscaling) is a CNCF graduated project that adds event-driven scaling to any Kubernetes workload. Instead of asking "how much CPU is this pod using?", KEDA asks "how many messages are waiting in this queue?". When there's work, it scales up. When there isn't, it scales down — all the way to zero if you want.

The architecture is clean. KEDA installs two components: a metrics adapter that exposes external signals as Kubernetes metrics, and a controller that manages ScaledObject and ScaledJob resources. You define what you want to scale and what signal should drive that scaling. KEDA handles the rest, including communicating with HPA under the hood — it doesn't replace HPA, it extends it.

Scalers: The Real Power

KEDA ships with over 60 built-in scalers. A few highlights that are immediately useful in Azure-heavy environments:

  • Azure Service Bus: Scale workers based on active message count or dead-letter queue depth. This alone solves a class of problems that used to require bespoke tooling.
  • Azure Storage Queue: Classic queue-depth scaling with minimal config.
  • Azure Event Hubs: Scale consumers based on partition lag — useful for high-throughput event pipelines.
  • Prometheus: Scale on any metric exposed in your Prometheus stack. Combined with OpenTelemetry, this means you can scale on business metrics, SLI signal, or anything you can instrument.
  • Cron: Schedule-based scaling for predictable load patterns — scale up before the morning batch job runs, not after.

ScaledObject vs ScaledJob

The distinction matters in practice. ScaledObject targets a running Deployment or StatefulSet and adjusts replica count continuously. Use this for long-running services that process work in a loop.

ScaledJob is different: it spins up a fresh Kubernetes Job for each unit of work and terminates it when done. This is the right model for batch processing — each message from a queue becomes its own isolated job with its own lifecycle. No shared state, clean retries, and your cluster only consumes resources when there's actual work to do. 

apiVersion: keda.sh/v1alpha1
kind: ScaledObject
metadata:
  name: worker-scaler
spec:
  scaleTargetRef:
    name: message-processor
  minReplicaCount: 0
  maxReplicaCount: 20
  triggers:
  - type: azure-servicebus
    metadata:
      queueName: processing-queue
      namespace: my-servicebus-ns
      messageCount: "5"  # target messages per replica

Scale-to-Zero Is the Killer Feature

HPA has a minimum replica floor — you can set it to 1, but not 0. KEDA removes that constraint. For event-driven workloads that only need to run when there's work (overnight batch jobs, webhook processors, scheduled data pipelines), this translates directly into cost savings. Pods that don't exist don't consume compute.

The cold-start question is real. If your consumers take 30 seconds to spin up, you might need to keep a warm replica. But for many workloads — especially those that can tolerate a few seconds of latency — scale-to-zero is viable and worth it.

Observability Considerations

KEDA exposes its own Prometheus metrics covering scaler errors, HPA replica changes, and trigger activity. Pair these with Grafana dashboards and you get visibility into scaling decisions in real time. It's also worth instrumenting your consumers with OpenTelemetry — tracking processing latency and throughput per replica gives you the data to tune messageCount targets intelligently rather than guessing.

When Not to Use KEDA

KEDA adds operational surface area. For services that scale fine on CPU/memory, the standard HPA is simpler and has fewer moving parts. Introduce KEDA when you have a clear event-driven signal that better predicts load than resource utilization — queue depth, lag, or schedule. Don't add it for its own sake.

That said, for teams running any kind of queue-based processing on Kubernetes, KEDA should be on the default platform stack. It's mature, well-maintained, and solves a real problem cleanly.
