MCP Servers: The Infrastructure Layer for AI Agents

2026-05-28
MCP AI Agents Architecture DevOps Infrastructure

Here's the problem with AI agents talking directly to APIs: they don't.

Instead, you build custom tool adapters for each agent. You write code to authenticate with Slack, code to call the GitHub API, code to query your database, code to hit the Kubernetes API. Each agent needs a slightly different version. You end up with a sprawl of tool definitions, auth handlers, and error-handling logic scattered across your codebase.

It's messy. It's fragile. It doesn't scale.

MCP (Model Context Protocol) solves this. Instead of agents calling APIs directly, they talk to MCP servers. The MCP server owns the authentication, API calling, error handling, and resource management. The agent just says what it wants and gets back structured data.

It's like SSH for AI. A standard, composable interface between agents and the systems they need to control.

What's an MCP Server?

An MCP server is a lightweight service that:

  • Exposes tools: Discrete, well-defined operations (list repos, create issue, query logs, etc.)
  • Owns authentication: API keys, OAuth, mTLS — the server handles it
  • Manages resources: Connection pooling, rate limiting, caching
  • Returns structured data: JSON schemas, predictable responses
  • Runs anywhere: Local process, container, Lambda, cloud VM — it's just a service

An MCP client (like an AI agent, Claude, or an agentic framework) connects to the server and asks: "What tools do you have?" The server responds with a manifest of available tools, their parameters, and expected outputs. The client can then invoke tools by name with the right arguments.

Example MCP server: GitHub 


MCP Server (GitHub):
  Tools:
    - list_repos(owner, filter)
    - create_issue(repo, title, body)
    - get_issue(repo, issue_number)
    - close_issue(repo, issue_number)
    - create_pr(repo, branch, title, body)
    - list_workflow_runs(repo)
    - trigger_workflow(repo, workflow_id, inputs)

Client (AI Agent):
  "I want to create an issue in my repo"
  → MCP Server handles: GitHub auth, API call, error handling
  ← Returns: issue ID, created_at, URL

  "Close issue #42"
  → MCP Server handles: auth, rate limiting, logging
  ← Returns: closed status, final state

Why This Matters

Without MCP, scaling agents is a nightmare:

Problem 1: Tool Duplication

You have 5 agents. Each one needs to talk to GitHub, Slack, and your PostgreSQL database. That's 15 custom tool implementations. Each one authenticates differently. Each one handles errors differently.

Six months later, GitHub changes their API. You update one tool. You forget to update the other 4. Now you have 3 agents that are broken.

With MCP: One GitHub server. All agents talk to it. Update once, all agents benefit.

Problem 2: Authentication Sprawl

Each agent has environment variables for credentials. GitHub token here, Slack token there, database password in a secret manager. You're managing N × M credential sets (N agents, M services).

With MCP: Credentials live in the MCP server. The agent just connects. No token management, no secret sprawl.

Problem 3: Resource Contention

Two agents try to hit the same API simultaneously. No rate limiting across agents. No connection pooling. No caching of expensive queries.

With MCP: The server manages rate limits, pools connections, caches results. All agents benefit from the same optimizations.

Problem 4: Observability

When something breaks, which agent called the API? What arguments? What was the response? You have to piece it together from logs across multiple services.

With MCP: All tool calls go through the server. Centralized logging, tracing, and debugging.

The MCP protocol itself is lightweight:

  • Initialization: Agent and server exchange capabilities
  • Tool discovery: "Here are my tools and their schemas"
  • Tool invocation: Agent calls a tool with arguments, server returns result
  • Streaming: Long-running operations can stream results back
  • Error handling: Structured error responses with actionable messages

Building an MCP Server

Simple example: GitHub MCP Server 


// Using the MCP SDK (Node.js example)
import { Server } from "@modelcontextprotocol/sdk/server/index.js";
import { StdioServerTransport } from "@modelcontextprotocol/sdk/server/stdio.js";
import { Octokit } from "@octokit/rest";

const server = new Server({
  name: "github",
  version: "1.0.0",
});

const octokit = new Octokit({
  auth: process.env.GITHUB_TOKEN,
});

// Define tools
server.setRequestHandler(ListToolsRequestSchema, async () => ({
  tools: [
    {
      name: "list_repos",
      description: "List repositories for the authenticated user",
      inputSchema: {
        type: "object",
        properties: {
          filter: { type: "string", description: "Filter by repo name" },
          limit: { type: "number", description: "Max repos to return" },
        },
      },
    },
    {
      name: "create_issue",
      description: "Create an issue in a repository",
      inputSchema: {
        type: "object",
        properties: {
          owner: { type: "string" },
          repo: { type: "string" },
          title: { type: "string" },
          body: { type: "string" },
        },
        required: ["owner", "repo", "title"],
      },
    },
  ],
}));

// Handle tool calls
server.setRequestHandler(CallToolRequestSchema, async (request) => {
  const { name, arguments: args } = request.params;

  if (name === "list_repos") {
    const repos = await octokit.repos.listForAuthenticatedUser();
    return { content: [{ type: "text", text: JSON.stringify(repos.data) }] };
  }

  if (name === "create_issue") {
    const issue = await octokit.issues.create({
      owner: args.owner,
      repo: args.repo,
      title: args.title,
      body: args.body || "",
    });
    return { content: [{ type: "text", text: JSON.stringify(issue.data) }] };
  }

  throw new Error(`Unknown tool: ${name}`);
});

// Start server
const transport = new StdioServerTransport();
await server.connect(transport);

That's it. You've got an MCP server that:

  • Exposes two tools: list_repos and create_issue
  • Handles GitHub authentication via env var
  • Converts tool calls into GitHub API calls
  • Returns structured JSON responses

Now any AI agent can connect to this server and use it.

The MCP Ecosystem

A few key servers are emerging:

  • GitHub MCP: Issues, PRs, workflows, repos
  • Slack MCP: Messages, channels, user management
  • Kubernetes MCP: Apply manifests, check pod status, logs
  • Database MCP: SQL queries, schema inspection
  • Filesystem MCP: Read/write/search files with safety guards
  • LLM MCP: Call Claude, GPT, Llama with model-specific parameters

The power is in composition. You can chain tools across servers:

"Find unresolved issues in my repos → For each, draft a fix → Create a PR → Post summary to Slack"

That's a multi-server workflow. Each MCP server owns its domain. The agent orchestrates.

Security & Sandboxing

One huge advantage of MCP: security.

Instead of agents having direct API access, they talk to a server. The server can:

  • Rate limit: Per-agent quotas on API calls
  • Audit: Every tool call is logged with who called it, what args, what happened
  • Gate: Some tools require approval before execution
  • Sandbox: Filesystem access is restricted to specific directories
  • Revoke: Disable a tool globally without touching agent code

This is critical when you have untrusted agents (third-party code, user-provided agents, experimental runs).

Practical Steps: Building Your First MCP Server

Step 1: Pick a domain. What system do your agents need to interact with most? Kubernetes? Slack? Your custom API?

Step 2: List the tools. "What are the top 5 operations agents need to do?" Make a list.

Step 3: Define schemas. For each tool, define inputs (parameters) and outputs (response structure).

Step 4: Implement handlers. Each tool maps to an API call or internal operation.

Step 5: Add safety. Rate limiting, logging, error handling, approval gates.

Step 6: Deploy. Run it as a service. Agents connect and use it.

You can build a basic MCP server in a few hours. A production-ready one with caching, retries, and observability takes a day or two.

The Future of Agent Integration

Right now, most agents call APIs directly or through custom wrappers. That works at small scale. But as you scale to 10+ agents, multiple teams, and dozens of integrations, it falls apart.

MCP is the infrastructure layer that makes scaling possible. It's not flashy. It's not the reasoning engine or the memory system. But it's the plumbing that ties everything together.

In the next few years, every mature agent platform will have an MCP server layer. It's how Kubernetes has ingress controllers, how cloud platforms have service meshes, how enterprise systems have API gateways.

It's not revolutionary. It's just better architecture.
