Platform Engineering × AI Agents: What's Actually Useful

2026-04-10
Platform Engineering AI Agents Kubernetes DevOps

Platform engineering teams are stretched thin. The promise was always "build the platform once, let developers self-serve." The reality is more like "build the platform, maintain the platform, debug the platform at 3 AM when someone's HPA config is wrong, and also write docs nobody reads." AI agents won't fix all of that. But they're starting to fix some of it in ways that actually matter.

The Toil Problem

Google's SRE book defined toil as "work tied to running a production service that tends to be manual, repetitive, automatable, tactical, devoid of enduring value, and that scales linearly as a service grows." Platform engineers live in toil. Triaging alerts, investigating why a deployment failed, explaining to the third team this week why their pods are in CrashLoopBackOff, updating runbooks that are already outdated by the time they're merged.

Traditional automation helps — you can script the known patterns. But toil is fractal. Every layer of automation reveals new toil underneath. The interesting thing about AI agents is that they can handle the unscripted parts: the investigation, the context gathering, the "let me look at six different things and synthesize an answer."

Where Agents Actually Help Today

After months of running AI agents as part of my own infrastructure workflow, here's what genuinely works:

1. Alert Triage and Context Gathering

An alert fires. The agent reads the alert, pulls the relevant pod logs, checks recent deployments, looks at resource utilization, and produces a summary: "Pod api-gateway-7f8b9 is OOMKilled. Memory usage spiked 3x after deployment v2.4.1 merged 2 hours ago. The deployment added an in-memory cache with no size limit. Here's the relevant diff." That's 15 minutes of human investigation done in seconds.

The key insight: the agent isn't fixing the problem. It's doing the boring, repetitive context-gathering that humans hate and are slow at. The human still decides what to do. But they decide with full context instead of starting from a cryptic PagerDuty message.

2. Runbook Generation and Maintenance

Nobody likes writing runbooks. Everyone agrees they're important. The compromise is usually runbooks that were accurate six months ago. An agent that watches incident channels, reads post-mortems, and observes how problems actually get resolved can draft and update runbooks continuously. Not perfect runbooks — but runbooks that reflect reality, which is more than most teams have.

3. Configuration Review

"Can you check if this Helm chart looks right?" is a question platform teams get daily. An agent that understands your cluster's conventions — resource limits, label standards, network policies, security contexts — can review configs before they hit the cluster. Not as a gate, but as a first pass that catches the obvious stuff: missing resource requests, privileged containers, services without health checks.

4. Documentation Q&A

The "how do I deploy to staging?" question. Every platform team answers it weekly. An agent with access to your docs, your CI configs, and your actual deployment history can answer it accurately and specifically: "Run make deploy ENV=staging. Your service uses the app-deploy pipeline which takes ~4 minutes. Last staging deploy was Tuesday by your teammate, here's the PR." That's better than any wiki page.

What Doesn't Work (Yet)

Autonomous remediation is still mostly hype. Yes, an agent can restart a pod. But should it? The blast radius of an agent that "fixes" things in production without understanding the full system context is terrifying. I've seen demos where an agent detects high CPU, scales up replicas, and declares victory — while ignoring that the high CPU was caused by a query of death that the additional replicas will happily amplify.

The problem isn't capability. Modern agents can run kubectl commands and modify resources. The problem is judgment. When a human SRE decides to scale up, they're drawing on years of context about the system, the team, the current deploy cycle, and the last five times this happened. An agent working from a 30-second context window doesn't have that. Not yet.

Cross-system reasoning is also shaky. An agent that can investigate within Kubernetes is useful. An agent that can trace a problem from the user's browser through the CDN, load balancer, service mesh, application code, and database — connecting the dots across seven different observability tools — is the dream. We're not there. Most agents lose coherence after three or four tool calls in a chain.

The MCP Connection

This is where MCP servers become essential. A platform engineering agent needs to talk to Kubernetes, your CI system, your monitoring stack, your incident management tool, your wiki, and probably Slack. Without a standard protocol, you're writing custom integrations for each one. With MCP, each tool gets a server, and the agent connects to all of them through the same interface.

The practical setup I've been running: MCP servers for kubectl operations, GitHub (PRs, issues, CI status), and log queries. The agent can go from "deployment failed" to "here's the failing test, the relevant code change, and a suggested fix" in a single conversation turn. Not because the agent is brilliant, but because it has access to the right tools through a consistent interface.

Patterns That Work

After iterating on this, a few patterns have emerged:

Human-in-the-loop by default. The agent investigates, summarizes, and recommends. The human approves. This isn't a limitation — it's the correct architecture for systems where mistakes cost money and trust.

Read-heavy, write-light. Let the agent read everything: logs, metrics, configs, docs, git history. Restrict writes to low-risk operations: creating issues, posting summaries, drafting PRs. The ratio should be 90% observation, 10% action.

Batch the toil. Instead of an agent that reacts to every alert in real time, have one that periodically sweeps through recent alerts, groups them by root cause, and produces a digest. "You had 47 alerts in the last 6 hours. 38 were the same intermittent DNS issue. 6 were related to the database migration. 3 need attention." That's more valuable than 47 individual investigations.

Make the agent's reasoning visible. When the agent says "this looks like an OOM issue," show the logs it read, the metrics it checked, and the chain of reasoning. Trust comes from transparency. An agent that says "trust me" is an agent that gets turned off.

Where This Is Going

The trajectory is clear: AI agents will become standard members of platform engineering teams. Not replacing humans — augmenting them. Handling the investigation grunt work, maintaining the documentation, answering the repetitive questions, and surfacing the patterns that humans miss because they're too busy firefighting.

The teams that figure this out first won't just be more efficient. They'll be more reliable, because their agents never forget to check the runbook, never skip the post-mortem action items, and never get too tired to read the logs carefully. The boring parts of platform engineering are exactly what agents are good at. Let them do it.
