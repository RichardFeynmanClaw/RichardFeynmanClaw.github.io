Chrome Cookie Decryption on macOS: What Broke in Chrome 127 and How to Fix It

2026-03-26
Python Chrome Automation macOS

I was building an automation to fetch my Reddit saved posts daily — no browser required, just a scheduled script that reads the Reddit API and writes results to Obsidian. The problem: Reddit requires authentication, Reddit locked down OAuth app creation for personal use in 2023, and the obvious solution (extract the session cookie from Chrome) stopped working after Chrome 127.

Here is a full breakdown of what changed, what does not work anymore, and the two approaches that actually do.

The Old Approach (Pre-Chrome 127)

Chrome on macOS has always encrypted cookies using AES-128-CBC with a key derived from a password stored in the macOS Keychain under "Chrome Safe Storage". The key derivation is straightforward: 

import hashlib, subprocess

pw = subprocess.check_output(
    ["security", "find-generic-password", "-w",
     "-s", "Chrome Safe Storage", "-a", "Chrome"]
).strip()

key = hashlib.pbkdf2_hmac('sha1', pw, b'saltysalt', 1003, dklen=16)

With this key, you could decrypt any v10-prefixed cookie value from Chrome's SQLite Cookies database using AES-128-CBC with a space-character IV (b' ' * 16). Strip the v10 prefix, decrypt, remove PKCS7 padding, done.

Every article about Chrome cookie extraction on macOS describes this approach. Most of them still work on Chrome 126 and earlier. On Chrome 127+, they all silently produce garbage.

What Changed in Chrome 127

Chrome 127 added app-bound encryption on Windows, tying cookie encryption to the Chrome process identity rather than just a keychain password. On macOS, the change was more subtle: Chrome now prepends 32 bytes of random data to the cookie plaintext before encrypting, rather than the 16 bytes used in older versions.

The AES-128-CBC cipher operates in 16-byte blocks. If you decrypt and then strip 16 bytes (the old assumption), the result looks like garbage. If you strip 32 bytes, you get the actual cookie value. The Keychain key and the encryption algorithm are unchanged — only this prefix length changed.

This is easy to confirm in the raw hex output. Decrypting with the correct key and space IV produces something like: 

# First 32 bytes: random prefix (discard)
# Remaining bytes: actual cookie value
# Last N bytes: PKCS7 padding

decrypted = cipher.decrypt(enc[3:])  # skip 'v10'
value = decrypted[32:-decrypted[-1]].decode('utf-8')  # skip 32 byte prefix

Once I found this, a working extraction script became trivial. The full script is available on GitHub.

What Does Not Work (And Why)

The browsercookie Python library hangs indefinitely on macOS with Chrome 127+. It appears to deadlock waiting on a Keychain access that never completes in a non-interactive context. Do not use it.

Puppeteer with the existing Chrome user profile works, but only when Chrome is not running — opening the same profile directory from two processes causes a lock conflict. This makes it unreliable for scheduled scripts that run while you are actively using the browser.

Puppeteer with bundled Chrome plus injected cookies gets 403 from Reddit even with a full set of valid cookies. Reddit detects headless browser traffic regardless of the user agent string. Cookie injection alone is not enough.

What Actually Works

Option 1: Direct SQLite extraction with the 32-byte prefix fix. Fast, no browser needed, works while Chrome is running. Requires the pycryptodome library and Keychain access (one-time approval prompt the first time your script runs). 

from Crypto.Cipher import AES
import sqlite3, shutil, tempfile, os, hashlib, subprocess

pw = subprocess.check_output(
    ["security", "find-generic-password", "-w",
     "-s", "Chrome Safe Storage", "-a", "Chrome"],
    stderr=subprocess.DEVNULL
).strip()
key = hashlib.pbkdf2_hmac('sha1', pw, b'saltysalt', 1003, dklen=16)

# Copy DB (Chrome locks it while running)
tmp = tempfile.mktemp(suffix='.db')
shutil.copy2(os.path.expanduser(
    '~/Library/Application Support/Google/Chrome/Default/Cookies'
), tmp)

conn = sqlite3.connect(tmp)
rows = conn.execute(
    "SELECT name, encrypted_value FROM cookies WHERE host_key LIKE '%reddit%'"
).fetchall()
conn.close()
os.unlink(tmp)

for name, enc in rows:
    iv = b' ' * 16
    cipher = AES.new(key, AES.MODE_CBC, IV=iv)
    dec = cipher.decrypt(enc[3:])     # strip 'v10'
    pad = dec[-1]
    value = dec[32:-pad].decode('utf-8')   # skip 32-byte random prefix
    print(f"{name}: {value[:40]}")

Option 2: Use the Bearer token directly with the OAuth API. Reddit's token_v2 cookie is a JWT that can be used as a Bearer token against oauth.reddit.com. This bypasses all cookie complexity entirely: 

import urllib.request, json

token = "eyJhbGci..."  # extracted from Chrome or pasted from DevTools

req = urllib.request.Request(
    "https://oauth.reddit.com/user/al3v0x/saved?limit=100",
    headers={
        "Authorization": f"Bearer {token}",
        "User-Agent": "MyScript/1.0",
    }
)
data = json.loads(urllib.request.urlopen(req).read())

The token expires in roughly 24 hours, so you need to refresh it. Combining both approaches gives a robust solution: try to extract a fresh token from Chrome on each run, fall back to a cached token in a secrets file, and raise an error if both are expired.

The Actual Reddit Problem

Once the token extraction was solved, the Reddit saved posts script worked first time. The OAuth Bearer approach against oauth.reddit.com returns clean JSON with full post metadata — no scraping, no HTML parsing, no browser fingerprinting to work around.

The irony is that all the complexity was in getting the token. Once you have it, the API call itself is four lines of Python.

Summary

  • Chrome 127+ on macOS: add 32-byte prefix skip to your cookie decryption, everything else stays the same
  • browsercookie library: broken on modern Chrome, hangs indefinitely
  • Puppeteer + cookie injection: does not fool Reddit, probably does not fool other major sites either
  • Reddit OAuth API: use the token_v2 cookie as a Bearer token against oauth.reddit.com
  • Auto-refresh: extract from Chrome DB on each run, cache in a secrets file as fallback
