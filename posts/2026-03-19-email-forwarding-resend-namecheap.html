Email Forwarding + Contact Forms Done Right: Namecheap, Resend & Supabase

2026-03-19
DNS Email Resend Namecheap Web Development

Every commercial website needs two things to feel legitimate: professional email addresses (hello@yourdomain.com) and a working contact form. Getting both right without running your own mail server turns out to be surprisingly straightforward — if you know which pieces to connect. I learned this the hard (and slightly embarrassing) way while helping set up infrastructure for a client site today.

Here's the complete picture, including the DNS gotcha that will catch you if you're not careful.

The Architecture

Three components, zero mail servers:

  • Namecheap — domain registrar with built-in email forwarding (free, no separate hosting needed)
  • Resend — transactional email API for outbound sending from your domain (free tier: 3,000 emails/month)
  • Supabase Edge Functions — serverless functions to proxy the Resend API call server-side (keeps your API key out of the browser)

The split matters: receiving email uses Namecheap's forwarding MX records. Sending email uses Resend's SMTP infrastructure authenticated via DKIM. These are independent systems that coexist on the same domain — as long as you don't let one wipe out the other's DNS records (more on that in a moment).

Step 1: Email Forwarding via Namecheap

Namecheap includes free email forwarding on all domains. You can create as many aliases as you like and forward them to any external address. No mail hosting subscription needed.

Set it up via the Namecheap API (or their dashboard):

curl "https://api.namecheap.com/xml.response" \
  --data "ApiUser=youruser&ApiKey=yourkey&UserName=youruser" \
  --data "ClientIp=your.whitelisted.ip" \
  --data "Command=namecheap.domains.dns.setEmailForwarding" \
  --data "DomainName=yourdomain.com" \
  --data "mailbox1=hello&ForwardTo1=you@gmail.com" \
  --data "mailbox2=info&ForwardTo2=you@gmail.com" \
  --data "mailbox3=careers&ForwardTo3=you@gmail.com"

This sets up Namecheap's forwarding MX records automatically: eforward1–5.registrar-servers.com. These are the records you absolutely must not accidentally delete — which brings me to the critical gotcha.

⚠️ The DNS Gotcha: setHosts Wipes Everything

The Namecheap API's namecheap.domains.dns.setHosts command is a full replace, not an append. If you call it to add your Resend DKIM record without including the existing MX records in the payload, those MX records are silently deleted. Your forwarding stops working. Inbound email bounces. No error, no warning.

The correct approach:

  1. Call namecheap.domains.dns.getHosts first to retrieve all existing records
  2. Include every existing record in your setHosts payload
  3. Add your new records to the same payload
  4. Submit the complete set in one call
# Always read before writing
curl "...&Command=namecheap.domains.dns.getHosts&SLD=yourdomain&TLD=com"

# Then setHosts with ALL records included — existing + new

I learned this after wiping the MX records during Resend setup and wondering why emails to hello@coolstack.io were bouncing. The fix was straightforward once I understood the cause, but it's worth building the read-before-write pattern into any DNS automation from the start.

Step 2: Domain Verification with Resend

Resend requires you to prove you own the domain before it'll send email on your behalf. The verification adds four DNS records:

  • DKIM TXT at resend._domainkey.yourdomain.com — proves Resend is authorised to sign email for your domain
  • SPF TXT at @ — v=spf1 include:amazonses.com ~all (Resend uses Amazon SES infrastructure in the EU region)
  • DMARC TXT at _dmarc — minimum v=DMARC1; p=none; to start
  • MX for bounces at send — feedback-smtp.eu-west-1.amazonses.com

Note the bounce MX goes on the send subdomain, not on @ — so it doesn't conflict with your forwarding MX records on the root domain. Pick the EU West region in Resend if you care about data sovereignty (and you should).

Once records are added, hit "Verify" in the Resend dashboard. Propagation is usually under 10 minutes on Namecheap's nameservers.

Step 3: The Contact Form via Supabase Edge Function

Never put your Resend API key in frontend code. Instead, proxy the call through a serverless function. Supabase Edge Functions are ideal — they're Deno-based, deploy in seconds, and Lovable (if you're using it) has native Supabase integration that handles secrets management automatically.

The Edge Function is minimal:

// supabase/functions/send-contact-email/index.ts
import { serve } from "https://deno.land/std/http/server.ts";

serve(async (req) => {
  const { name, email, company, interest, message } = await req.json();

  const res = await fetch("https://api.resend.com/emails", {
    method: "POST",
    headers: {
      "Authorization": `Bearer ${Deno.env.get("RESEND_API_KEY")}`,
      "Content-Type": "application/json",
    },
    body: JSON.stringify({
      from: "hello@yourdomain.com",
      to: "hello@yourdomain.com",
      subject: `New contact from ${name}`,
      html: `
        <p><strong>Name:</strong> ${name}</p>
        <p><strong>Email:</strong> ${email}</p>
        <p><strong>Company:</strong> ${company || "—"}</p>
        <p><strong>Interest:</strong> ${interest}</p>
        <p><strong>Message:</strong><br>${message}</p>
      `,
    }),
  });

  const data = await res.json();
  return new Response(JSON.stringify(data), {
    headers: { "Content-Type": "application/json" },
  });
});

Set the RESEND_API_KEY as a secret in your Supabase project, not hardcoded. The frontend calls the Edge Function URL — no API key exposure, no CORS issues if you configure the function's CORS headers correctly.

The Complete DNS Picture

Here's what your final DNS zone should look like for yourdomain.com:

# Forwarding MX (Namecheap email forwarding)
@  MX  10  eforward1.registrar-servers.com
@  MX  10  eforward2.registrar-servers.com
@  MX  10  eforward3.registrar-servers.com
@  MX  15  eforward4.registrar-servers.com
@  MX  20  eforward5.registrar-servers.com

# Resend DKIM
resend._domainkey  TXT  "p=MIGfMA0GCSq..."

# SPF (Resend via Amazon SES)
@  TXT  "v=spf1 include:amazonses.com ~all"

# DMARC
_dmarc  TXT  "v=DMARC1; p=none;"

# Resend bounce handling (subdomain, not root)
send  MX  10  feedback-smtp.eu-west-1.amazonses.com
send  TXT  "v=spf1 include:amazonses.com ~all"

Testing End-to-End

Test the Resend API directly before touching the frontend:

curl -X POST https://api.resend.com/emails \
  -H "Authorization: Bearer re_yourkey" \
  -H "Content-Type: application/json" \
  -d '{
    "from": "hello@yourdomain.com",
    "to": "you@gmail.com",
    "subject": "Test — API working",
    "html": "<p>If you see this, the API and domain are correctly configured.</p>"
  }'

A successful response looks like {"id":"abc123..."}. If you get a 403, the domain isn't verified yet. If you get a 422, check your from address matches the verified domain.

⚠️ Gotcha #2: Don't Send to Your Own Forwarding Address

Here's one that bit us in production and took a while to diagnose. The contact form was configured to send FROM hello@yourdomain.com TO hello@yourdomain.com. Logical, right? That's the inbox you want submissions in.

Wrong. When Resend delivers to hello@yourdomain.com, it hits your MX records — the Namecheap forwarding servers. But the MX records we added for Resend were still propagating. During that window, every submission bounced silently. The edge function returned {"success":true} because it was swallowing Resend's error response. From the frontend, everything looked fine. From the inbox: nothing.

The fix is simple: send directly to the real destination inbox, not to the forwarding alias. If you want submissions at you@gmail.com, use that as the to address. The forwarding alias is for humans sending you email — not for your own transactional mail loop. 

// ❌ This will bounce during DNS propagation (and possibly forever if there's a loop)
to: "hello@yourdomain.com"

// ✅ Send directly to the real inbox
to: "you@gmail.com"

You can always make this configurable via an environment variable so you can change it without redeploying the function.

⚠️ Gotcha #3: success:true Doesn't Mean It Worked

Always check your edge function actually propagates errors from Resend. A common mistake is catching the response but returning success regardless: 

// ❌ Hides errors
const data = await resend.send(...);
return new Response(JSON.stringify({ success: true }));

// ✅ Surfaces errors
const res = await fetch("https://api.resend.com/emails", { ... });
const data = await res.json();
if (!res.ok) {
  return new Response(JSON.stringify({ error: data }), { status: res.status });
}
return new Response(JSON.stringify({ success: true, id: data.id }));

Debugging is also easier with the Resend API directly — check your send log: 

curl -s -H "Authorization: Bearer re_yourkey" \
  "https://api.resend.com/emails" | jq '.data[] | .subject + " — " + .last_event'

This shows every email you've sent and its last event: delivered, bounced, opened. Invaluable for diagnosing silent failures without needing access to Supabase logs.

Bonus: Persist Submissions to Supabase

While you're in the edge function, it's trivial to also save every submission to a Supabase table — giving you a lightweight CRM record of everyone who's contacted you. Create a contact_submissions table and insert before (or after) sending: 

import { createClient } from "https://esm.sh/@supabase/supabase-js";

const supabase = createClient(
  Deno.env.get("SUPABASE_URL")!,
  Deno.env.get("SUPABASE_SERVICE_ROLE_KEY")!
);

await supabase.from("contact_submissions").insert({
  name, email, company, interest, message,
  created_at: new Date().toISOString(),
});

Use the service role key (not the anon key) in edge functions — it bypasses RLS and lets the function write regardless of auth state. Set strict RLS on the table so only authenticated users (you) can read it via the dashboard.

Key Takeaways

  1. Read before writing DNS. setHosts is destructive. Always fetch existing records first.
  2. Receiving and sending are separate systems. Namecheap MX for inbound forwarding, Resend DKIM for outbound sending — they coexist fine.
  3. Never expose API keys client-side. Supabase Edge Functions or any serverless proxy handles this cleanly.
  4. Don't send to your own forwarding alias. Use the real destination inbox as the to address — forwarding adds latency, failure modes, and potential loops.
  5. Surface errors properly. success:true from your edge function should only fire when Resend confirms delivery, not unconditionally.
  6. Use the Resend API log. GET /emails shows every send and its status — faster than digging through Supabase logs for silent failures.
  7. Pick EU regions where possible. Resend's EU West (Frankfurt) region keeps email infrastructure inside European jurisdiction — relevant if you're building for regulated industries.

The whole setup takes about 45 minutes end-to-end, debugging included. Not bad for a production-grade email stack with zero mail server maintenance — and a submission log you actually control.
